Timesheet Day View Revamping
by Daniel Marino

View Prototype

I greatly enjoyed the challenge of reworking an interface that I use almost daily: the Timesheet Day View. I spent around 15 hours in total on this, working on sketches/wireframes, some Photoshop comps, and a semi-working prototype. Below I will detail my process.

Sketches/Wireframes

I find sketching really helpful for two reasons: One, it helps get ideas out on paper quickly, and two, it gives a great starting point for discussing ideas with other team members. Sketching can prevent a lot of wasted time by finding pain points in ideas before going straight to high-fidelity mockups or prototyping/development.

Based on discussions with Shawn, some of the customer issues we are trying to solve are:

  1. Making input fields less ambiguous to the user on the time entry form - particularly the task and hours input fields.
  2. Allowing users to enter and edit times without hiding existing entries. It appears customers like to be able to see past entries for the day while entering new entries for context.

Time Entry Dialog

I feel that Harvest's Mac App does a decent job of handling the Time Entry dialog, so I took some cues from that interface. I used the placeholder attribute to describe the project, task and note input fields (and use <label> elements as a graceful fallback for older browsers that don't support the placeholder attribute). By hiding <label> elements each input field flows nicely into the next. It also reduces the overall height of the time entry form by about 30%. Double-win!

Shawn said that users were particularly vexed by the hour input field because there was no labeling, and using a decimal value as a placeholder threw users off. To fix this I added a clock icon to give the user some context for what kind of data should be entered into the input field, as well as descriptive text (borrowed from the Mac App) to the right of the input field.

I wrestled with the decision for a little as to whether the placeholder should read "0.00" (with a decimal) or "0:00" (with a colon). I feel that using a colon will better communicate that the input is looking for time-related data. However, the value is immediately converted to a decimal number with JavaScript, and I feel this would confuse users even more. Thus, in the end I stuck with the decimal format. Having said all that, I think this would make an interesting user test.

Adding/Editing Entries

Similar to a previous version of the Timesheet Day View page, I think it makes sense to append the time entry dialog to the bottom of existing entries. It provides proper context chronologically and allows the user to review previous entries while entering a new one.

It also makes sense to allow the user to edit existing entries where they already exist in the page.

The Nitty Gritty (tools & libraries used)

I love reading how websites get built. In case you do too here are the tools and libraries I used:

  • SASS/Compass for writing and compiling CSS
